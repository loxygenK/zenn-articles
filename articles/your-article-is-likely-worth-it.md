私があまり記事を書いていないことにはいろいろな言い訳があるのですが、メジャーな言い訳の一つとして、こういうのがありました:

**自分が学んだことって、他人から学んでることだし、、じゃあ自分が記事を書いても同じことがありふれちゃうだけじゃない？？？**
- → 特別、質のいい記事を書けるわけでもないし、わざわざ書いてなんかあるんだろうか、、
- → 自分に対してのメリット（知識の体系化）とかは、自分用 Obsidian で済ませればいいし、そのためにわざわざインターネットを汚したくはない、、

たまたま、技術記事を書きなさい、ということに関するコンテンツに触れていて、この言い訳に関して気づくことがあったので、早速書いてみようということで、書いてみます!

## 本記事のスコープ

「自分が持っている知識は、すでに誰かの知識…… それについて新しく記事を書くことに意味はある?」ということについて、「**知識のユニークさは求められなくて、知識と自身の経験の組み合わせが大事**」という観点から噛み砕いでみる記事です！「初心者の記事にこそ意味がある」という理由にもにょってる人にもおすすめです

#### 前提としていること

「インターネットを汚したくない」という良心があり、記事を執筆するにあたって、**事実確認やレビューは通す ー つまり、正確性は（自分のできる範囲で）保証している**ことを前提とします。
つまりこの記事は、**執筆を開始するにあたって、他人の記事の存在は一旦気にしなくても大丈夫**ということを述べていて、**記事を量産してもいいんじゃない?? ということは述べていません**!

#### 書かないこと

記事を書くことについては、この記事で触れている理由以外にも、自分/コミュニティ両者に対する利点が多くあります。ただ、この記事では、この理由にだけフォーカスするので、他の理由にはあまり言及しません。

## 結論：知識のユニークさより「経験＋解釈」

最初に結論を書いておきます。

*同じ知識についての記事がたくさんあってもしょうがなくない?*

- **知識そのもののユニークさは、そもそも二の次・三の次**
- 知識だけを見せつけられても、**人間はなかなか共感できないし、感情的にも惹きつけられず**、何かと身につきにくい
- 誰かの経験を踏まえた解釈といった、**「人間の温かみ」が乗ることで、初めて知識に共感しやすくなり**、腹落ちしやすくなる
- 一方で、どんな経験に共感できるかは**読者によってバラバラ**なので、予測するのはむずい
- → だから、**同じ知識について複数の記事が存在していても、それぞれに解釈がある限り、むしろ良いこと**

*自分が書いた記事が新しく価値を生むとは思えない*

- 盗作とか、AI に全てを任せたとかでない限り、**自分が得た知識には必ず「自分の解釈」がくっついている**
- 自分は他人とは別個体なので、**自分の人生や経験は唯一無二**
- → だから、**特別「尖ったこと」をしなくても、結果的に唯一無二の解釈になりやすい**

まとめると、「自分で記事を書く」という行為をしている限り、それは誰かが知識に共感できるようになるための「唯一無二の解釈」を公開している、ということになるはずです。

複数の解釈が並んでいること自体が、**どこかの誰かが、その知識に共感できるかもしれない解釈を増やすことができる良いこと**[^neutral-is-good-sometimes]であり、同じ知識についての記事であっても、新しい解釈を提供するのはポジティブなことだと思うようになりました。
[^neutral-is-good-sometimes]: 自分の解釈が入っていることが常に良いこと、とも限らないですが - これに関しては後述しています!

そして、その新しい解釈は、実は特別意識しなくても自然とにじみ出てくるはずです。

## 具体例：lockfile の話で考えてみる

もう少し具体的に考えるために、次のような技術ネタを例にします。

```
lockfile (pnpm-lock.yaml) って結局コミットすべき?
```

lockfile は自動生成されるファイルなので、最初のころに一度は迷うポイントだと思います。

Stack Overflow にもかなり upvote のついている Q&A があります（pnpm じゃなくて npm で恐縮ですが、、）:

> …… Is this file supposed to be kept in source control? ……
> [node.js - Do I commit the package-lock.json file created by npm 5? - Stack Overflow](https://stackoverflow.com/questions/44206782/do-i-commit-the-package-lock-json-file-created-by-npm-5)

`pnpm-lock.yaml` をコミットすべきかに関しては、公式ドキュメントではっきり「コミットしてね」と書かれています。

> You should always commit the lockfile …… (以下略)
> [Working with Git | pnpm](https://pnpm.io/git)

つまり、**「lockfile はコミットするべき」という知識自体は、そんなにレアな情報ではなく、おそらくいろいろな記事がある**と推測できます。

## 深掘り 1：同じ知識でも、「たどり着き方」の違いが価値になる

ここで一度、「同じ知識に言及するたくさんの記事は、果たして必要なのか？」という問いに戻ります。

この問いは、少なくとも次の 2 つに具体化できそうです。

1. **全部同じ知識について述べているんだから、全部似たようなものじゃない??**
2. **似たような経験をしている人がいそうだから、記事も似るのでは？**

1 つ目に関しては、図にマッピングすると「**最短距離の矢印が至高で、他はしゃあないだけでは？**」という見方ができます。

![先程の図と似ていますが、最短距離の 1 本を強調して、「最短距離が正義?」と書いてあります](/images/your-article-is-likely-worth-it/shortest-might-be-the-best.png)

- 最短距離の矢印: 余計な情報はなく、「lockfile はコミットしよう」という**知識だけがダイレクトに伝わる**
- 寄り道の矢印たち: **個人的な体験談や気づきを噛みながら**、同じ知識にたどり着く

ここで、記事を読んだ人が、どのように感じるかを考えてみます。

![先程の図と似ていますが、先程の図での、最短の矢印への強調は解除されています。それぞれの矢印に対して、読者が反応を示しています。まず、最短距離の矢印に対しては、「なんつーか、そうなんだけど…」とリアクションしています。上の矢印には、「えぐい、やっぱり必要なんだなぁ」とリアクションをしています。下の矢印には、「あー、そういう視点もあるのか」とリアクションをしています。](/images/your-article-is-likely-worth-it/possible-reaction-by-reader.png)

- 最短距離の矢印
    - 自身の経験に紐付けられるフックがなく、**納得感は薄くなりがち**
        - 「なんつーか、そうなんだけど、、」とまだモヤりそう
        - 「そうなんだ、覚えておこう[^they-are-not-understanding-actually]」で終わるかもしれない[^might-be-ok-if-reader-is-good]
- 失敗談を通る矢印（上の方）
    - 「えぐ、そんなことになるの？」と感情が少なからず動く
    - **執筆者のエピソードや解釈に乗せて知識を理解できる**
- セキュリティの矢印（下の方）
    - 「lockfile がないとそうなるのか、新しい視点だ」と感じられる

[^they-are-not-understanding-actually]: "覚えておこう" という、(理解はしていないけど) とりあえず覚えておこう・・となっているのがミソです

[^might-be-ok-if-reader-is-good]:
    ところで、ある程度慣れているなら、すでに自分の中で経験が多くあり、インデックス化も済んでいるかと思われます。
    こういう状態だと、公式ドキュメントを読むなど、特に寄り道なしで直接知識を受けることに関して、特に抵抗はないかと思います。一方で、初心者の方だったり、あるいは別分野で経験はあるけど、新しい分野に飛び込んでいる方など、必ずしも知識をそのまま受けても、それを自身の経験と関連づけられず、共感までいけない、、ということがあるかと思います

どの寄り道にも共通するのは、「知識を得る」という点ではある種の“無駄”がある一方で、**他人の経験に共感することで、感情も動員しながら知識に触れられる**、ということです。

ここまでをまとめると、

- **知識だけをポンと出されても、それを身につけるのは案外大変**
- ですが、**誰かの経験や感情のフックがあると、その知識に感情的にアタッチしやすくなる**

という感じです。

また、その「どのフックに刺さるか」は**読者の数だけバリエーションがある**かと思います。全員別個体の人間ですから、各々別々の人生 / 別々のキャリアを歩んでいるはずです。**読者一人一人が違う経験を持っているなら、刺さるフック（共感できる経験）は人の数ほどバリエーションがある**はずです。

といいつつ、1 つの記事で多くのフックを持たせようとすると労力がかかりますし、記事の軸もぼやけてしまいます。だからこそ、同じ知識に言及はしていれど、**違うフックを持った複数の記事が並存している**ことは、**それぞれの記事は自分の経験についてフォーカスしつつ、バリエーション豊かなフックが準備されている**点で、とても自然で健全な状態なのでは、と感じます。

## 深掘り 2：似たような経験でも、細部まで一致することはほぼない

これまで、「**知識は同じでよくて、バックグラウンドになってる経験の違いが価値を生む**」ということについて論じましたが、**似たような経験をしている人がいそう**という点についてはどうでしょうか。

先ほど、「チームメンバーの PC で違うバージョンが入っていた」という失敗談をひとつ例にしましたが、今度はこういうケースを考えてみます。

- チーム開発ではなく、個人開発プロジェクト
- CI を回してみたところ、**CI 環境でだけ違うバージョンが入ってしまった**
- その結果として、この人も「環境間の冪等性」の大切さに気づいた

![先程の図と同じですが、上の矢印とほぼ同じ経路を辿る矢印が増えました](/images/your-article-is-likely-worth-it/two-similiar-articles.png)

どちらの記事も、

- 個人的な失敗であること
- 環境間の冪等性について認識したこと

という点では「似ているなぁ……」と感じられます。

ただ、実際には次のような違いがあり得ます。

![先程の図で重複した 2 つの矢印にフォーカスを当てた画像です。それぞれの矢印が経由する経験は、先程述べた 2 点のような共通事項はありつつも、後述するような差異が明らかに存在し、2 つの矢印の間で違うものになっています](/images/your-article-is-likely-worth-it/everything-is-hardly-duplication-here.png)

- もともとあった、チームメンバー側の記事
    - 実際にメンバーの PC でコマンドを一緒に叩いたり、差分を調べたりしたプロセスが描ける
- CI 側の記事
    - CI のログを読む試行錯誤や、再実行のたびに時間やリソースが溶けていくつらみを書くかもしれない
- それぞれで遭遇したエラーメッセージも、おそらく違う!

つまり、ざっくり見れば似たような経験でも、**細部を見れば完全に一致することはほぼありません**。

さらに言えば、先程述べたように、私たちは全員それぞれ違うキャリアパスを持っています。仮に似通っている人はいても、完全一致はほぼないはずです。だから、**自分の経験や考えを踏まえて**書けば、自ずと**他人とは違う記事 - 新たなバリエーションを増やせる**んじゃないかと感じます。

## 「自分の経験を踏まえて書く」のはむずい?

「**自分の経験を踏まえて書けば、自然と新しい価値になるはず**」という話を書いてきましたが、執筆時に「自分の経験を書くぞ！！」と意気込む必要はあるでしょうか?

私は、ここについてはあまり心配していません。どんな事実にしても、私たちがそれを知覚する時には、**必ず自分の経験を踏まえて解釈・知覚しているはず**だからです。分解すると…

- ピュアな「事実だけ」を知覚するのは、人間にはほぼ無理
- 新しく知識を理解して「あぁ〜〜そういうことか」となったとき …
- … その裏側では、**自分の経験やすでに持っている知識と関連づけるプロセスが走っている**[^sounds-like-ai]

… という感じです。ゆえに、人間の性質的に「**自分の言葉・解釈で書く**」と、自ずと**自分の経験を踏まえた文章になるだろう**、ということです！

逆に、「自分の経験を踏まえずに書く」ことになりそうなのは、例えば次のようなケースです。

- ドキュメントの丸写し（自分の解釈を一切入れない）
- AI に最初から全部書かせてしまい、自分の経験を挿入しない

AI に文章を書かせること自体を否定したいわけではありません[^written-by-ai]。ただ、「自分の経験を入れ込まずに全部 AI に任せる」と、おそらく経験を踏まえていない文章になりそう、、という話です。

また、チーム開発のドキュメントのように、あえて個人色を消した方が良い場面もあると思います。一方で、不特定多数の人に読んでもらう記事であれば、やはり自分の経験があった方が、共感を得られやすく、より読者にとって価値のあるものになりそうだと感じます。

## まとめ：自分が記事を書くことには意味がありそう

- **知識自体のユニークさは、二の次・三の次**
    - 同じ知識に言及しているからといって、それだけで「しょうもない記事」とは言えない
- 個人の経験は、**執筆のときにずっとそばにいてくれる**[^sounds-like-ai]
    - 人が自分で得た知識は、必ず自分の経験を踏まえて解釈・知覚されている
    - 自分の人生・経験は唯一無二なので、盗作でもない限り、**意図せず完全に被ることは無視できるくらい可能性が低い**（もし被ったらそれはそれで面白い）
- さまざまな知覚・解釈があることは良いこと
    - 知識だけを見せられても「そうだね」で終わりがち
        - 合理的な情報だけをロスレスに吸収するのはむずかしい
    - ちょっとした人間の温かみがあるだけで、**情報に共感しやすくなり、感情的にもアタッチしやすくなる**

読者が共感できる「個人の経験」は人によって違いますし、人の数だけパターンがあります。それを事前に全部予測するのは不可能です。だからこそ、「自分が記事を書くことには意味がないのでは？」と感じていたとしても、**経験を踏まえて自分の言葉で書いた時点で、それは誰かにとって新しいフックになる**[^sounds-like-ai]可能性が高いはずです。
